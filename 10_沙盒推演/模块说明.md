# 10_沙盒推演 — 模块说明

> 零系统的"风洞实验室"。
> 在真实执行前，先在沙盒里跑一遍，提前发现会撞的墙。

---

## 一句话定义

**沙盒推演 = 在脑子里跑一遍系统未来 1-3 个月的运行，提前识别瓶颈、风险和优化点。**

它不是在预测未来——未来不可预测。它是在提前发现**已知模式 + 当前约束 + 计划行动**三者交叉时，会在哪里出问题。

---

## 为什么需要这个模块

零系统的假设是"先跑起来再调整"。但有些问题不需要亲身踩进去才知道：

- 少爷的已知模式是第 4 周崩溃——不需要再验证一次
- 同时启动三条线消耗是单线的 3 倍——不需要等到 Week 4 才发现
- 周末 8h 学习剥夺恢复时间——不需要等到连续疲劳才调整

**沙盒推演的作用：把"试错成本"从真实生活转移到文档里。**

## 模块结构

```
10_沙盒推演/
├── 模块说明.md          ← 本文件。模块概述
├── 模拟方法论.md        ← 推演框架：输入、假设、分析方法、迭代机制
├── 推演情景库.md        ← 可复用的情景模板 + 当前已知情景
└── 推演记录/
    ├── V1_2026-05-21.md ← 第 1 次推演：单线身体系统
    └── V2_2026-05-21.md ← 第 2 次推演：三线系统（当前）
```

## 与其他模块的关系

| 模块 | 关系 |
|------|------|
| **00_系统核心/行为模式.json** | 推演的输入。已知模式决定推演的假设条件 |
| **00_系统核心/当前状态.json** | 推演的起点。当前数值决定初始状态 |
| **09_自我进化/优化建议池** | 推演的输出。发现的优化点写入建议池 |
| **09_自我进化/迭代路线图** | 推演的输出。识别出的待办任务写入路线图 |
| **04_交易方案 + 05_身体系统** | 推演的对象。推演结论反馈回这两个模块调整方案 |
| **02_目标系统/** | 推演的验证。推演发现目标不合理时反馈调整 |

---

## 什么时候运行推演

| 触发条件 | 示例 | 输出 |
|----------|------|------|
| 系统重大变更 | 新增支柱、修改方案、变更身份 | 完整的 Vx 推演报告 |
| 执行瓶颈出现 | Week 3-4 连续中断、疲劳不降 | 专项推演 + 优化建议 |
| 月度复盘前 | 每 30 天一次的例行推演 | 简版推演 + 目标调整建议 |
| 用户主动要求 | "推演一下接下来会怎样" | 按用户指定的时间范围和情景 |

---

## 推演产出物

每次推演输出三个东西：

1. **推演报告** — 写入 `推演记录/`，完整记录推演过程、结论、风险清单
2. **优化建议** — 发现的优化点写入 `09_自我进化/优化建议池.md`
3. **迭代任务** — 确认要做的事情写入 `09_自我进化/迭代路线图.md`

---

## 原则

| 原则 | 含义 |
|------|------|
| 不预测，只识别 | 推演不是算命，是识别已知模式 + 当前约束下的可能性 |
| 推演必须可验证 | 每次推演都记录假设条件，3 个月后回头看哪些对了哪些错了 |
| 推演永不复用旧假设 | 每次推演前重新读取当前状态和最新已知模式 |
| 推演结果必须产生行动 | 没有行动建议的推演是无效的 |
