## 1. 项目概述

### 1.1 项目定位

「零」是一个以文件系统为持久化媒介的**个人人生目标操作系统**。它不是一个传统的业务应用程序，而是由**规则文档**、**状态 JSON**、**目标规划文档**、**每日执行记录**和**仪表盘看板**共同组成的个人运营系统。

### 1.2 核心理念

- 长期主义优先：宁可慢，不要断
- 稳定执行 > 短期爆发
- 动态调整 > 固定计划
- 可持续 > 理想化
- 减少混乱与内耗，防止系统崩溃

### 1.3 两大长期目标

| 优先级 | 目标 | 描述 |
|--------|------|------|
| 1 | 身体巅峰 | 街健美学体型，精瘦有力量感，上镜好看 |
| 2 | 财务自由 | 建立被动收入系统，摆脱纯打工状态 |

### 1.4 系统运行闭环

```
读取核心档案 → 判断当前状态 → 结合目标与规则制定今日策略 → 记录执行 → 复盘与动态调整
        ↑                                                                      |
        └──────────────────────────────────────────────────────────────────────┘
```

## 2. 运行方式与数据形态

### 2.1 核心运行机制

**「零」系统的运行不依赖任何应用程序——它的运行时就是与 AI 的对话。**

系统的工作方式如下：

```
用户 → 把「系统提示词」作为提示词交给 AI → AI 读取核心文件 → 在对话中完成分析/调度/记录/复盘
```

| 角色 | 职责 |
|------|------|
| AI | 运行时引擎：读取文件、分析状态、制定策略、记录观察 |
| 用户 | 执行者 + 反馈者：执行任务、报告进度、更新档案 |
| 文件系统 | 持久化记忆：存储所有状态、目标、日志、规则 |

### 2.2 入口：系统提示词

系统的真正入口是 `00_系统核心/系统提示词.md`。每次新对话时将其作为 AI 的提示词，触发完整的启动流程。

### 2.3 项目中代码的定位

`src/` 目录下的 `.js` 文件和 `package.json`、`push.bat` 均为**可选的辅助工具**，不参与系统核心运行：

| 工具 | 用途 | 是否必需 |
|------|------|----------|
| `src/index.js` | 终端状态查看器 | 否，纯展示用 |
| `src/generate_dashboard.js` | 仪表盘自动生成 | 否，辅助维护 |
| `src/push.js` | Git 推送（跨平台版） | 否，替代 push.bat |
| `push.bat` | 一键 Git 推送 | 否，方便同步 GitHub |

### 2.4 项目数据形态

| 格式 | 用途 | 示例 |
|------|------|------|
| `.md` | 规则文档、目标文档、执行日志、仪表盘 | `系统提示词.md`, `2026-05-21.md` |
| `.json` | 核心状态持久化、结构化数据 | `当前状态.json`, `人格与方向.json` |
| `.js` | 辅助脚本（可选） | `src/index.js` |
| `.bat` | 一键 Git 推送脚本 | `push.bat` |
| `.gitkeep` | 占位符（预留空目录） | 交易系统等模块目录 |

## 4. 模块详解

### 4.1 00_系统核心 — 控制层

> 存放系统规则、AI 行为约束、系统提示词和运行流程。是整个系统的"大脑"。

#### 文件清单

| 文件 | 职责 | 核心内容 |
|------|------|----------|
| `AI系统自身概要.md` | AI 的系统身份卡，启动时首先读取 | 系统身份、架构图、启动流程、运行规则、历史里程碑、用户档案快照 |
| `系统提示词.md` | 定义 AI 身份与行为规范 | 身份定义、工作模式（目标分析/动态调整/最低执行）、输出风格、启动流程 |
| `运行规则.md` | 目录结构与核心文件职责定义 | 9 个目录结构、6 个 JSON 职责、复盘节奏、版本控制规则 |
| `AI行为规则.md` | 行为约束 | 禁止复杂化、禁止空谈、动态调整、风险优先、对话规则、记忆规则 |
| `工作流程.md` | 每日/每周/每月操作流程 | 启动→调度→执行→复盘 四阶段流程 |

#### 关键概念

**启动流程（7 步骤，每次新对话必执行）**：

```
第1步：读取 00_系统核心/AI系统自身概要.md — 恢复系统身份、运行规则、历史里程碑
第2步：读取 01_人生核心档案/ 下全部 6 个 JSON — 恢复持久化记忆
第3步：读取 08_仪表盘/ 下的 Markdown 文件 — 快速了解当前状态
第4步：分析当前状态（精力 / 动量 / 纪律 / 疲劳 / 压力）
第5步：检查是否有未处理的 AI 观察
第6步：检查当前阶段和优先级
第7步：确定今日执行策略
```

**工作模式**：

| 模式 | 触发条件 | 行为 |
|------|----------|------|
| 目标分析模式 | 用户提出新目标 | 通过提问深度理解 → 理解达 90% 后制定计划 |
| 动态调整模式 | 常态运行 | 根据执行率/情绪/精力/身体恢复动态调整 |
| 最低执行模式 | 连续摆烂/熬夜/执行断裂/严重疲劳 | 只保留最核心任务 + 身体恢复 + 睡眠 + 最低行动量 |

### 4.2 01_人生核心档案 — 记忆层

> 系统的持久化记忆核心。6 个 JSON 文件共同构成 AI 每次启动时需要恢复的完整上下文。

#### 文件清单

##### 人格与方向.json

```json
{
  "人生方向": ["身体巅峰", "财务自由"],
  "核心价值观": ["长期主义", "掌控感", "自由", "自律"],
  "性格特征": {"类型": "完美主义倾向", "风险": "容易被过量压垮"},
  "身体档案": {
    "身高": 175, "体重": "140斤",
    "旧伤": ["肩伤史", "肩峰撞击风险"],
    "体态问题": ["脖子前倾", "骨盆前倾"],
    "当前能力": {"引体向上": 8, "俯卧撑": 20}
  },
  "工作档案": {"职业": "程序员", "工资": "8.5k", "工作时间": "9:00-12:00, 13:30-18:00"},
  "睡眠模式": {"当前": "1~2点睡", "风险": "长期熬夜"}
}
```

**职责**：系统最核心的身份定义文件。永不频繁修改，仅重大认知升级时更新。

**关键字段**：
- `人生方向`：长期追求的终极方向
- `性格特征.类型`：完美主义倾向——系统所有防护措施的核心依据
- `身体档案.旧伤`：肩伤史和肩峰撞击风险——决定了训练计划的安全底线
- `工作档案`：7.5 小时工作制，周末 2 天——影响可支配时间规划

##### 长期目标.json

```json
{
  "长期目标": [{ "目标": "身体巅峰", "优先级": 1 }, { "目标": "财务自由", "优先级": 2 }],
  "当前阶段": "Phase 1: 重建掌控期",
  "阶段描述": "混乱度10/10，身体1/10。先重建最小自律系统。",

    { "目标": "建立每日训练节奏", "状态": "进行中" },

  "优先行动": ["每天4个核心动作", "禁止过量训练、熬夜后高强度、肩痛硬顶"]
}
```

**职责**：存储目标的层级结构——长期目标 → 当前阶段 → 季度目标 → 优先行动。

**更新时机**：阶段切换时更新。

##### 当前状态.json

```json
{
  "精力": 3, "动量": 3, "纪律": 3,
  "疲劳": 7, "压力": 6,
  "阶段": "Phase 1: 重建掌控期",
  "最后更新": "2026-05-20T20:30:00.000Z"
}
```

**职责**：5 维实时状态快照（精力/动量/纪律/疲劳/压力，1-10 分制）。每次对话结束时更新。

**字段含义**：

| 字段 | 含义 | 当前值 | 解读 |
|------|------|--------|------|
| 精力 | 体能与精神状态 | 3/10 | 偏低，训练量不能大 |
| 动量 | 连续执行的惯性 | 3/10 | 刚启动，需要累积 |
| 纪律 | 自律稳定度 | 3/10 | 脆弱，易断裂 |
| 疲劳 | 身体累积疲劳 | 7/10 | 偏高，需关注恢复 |
| 压力 | 心理压力 | 6/10 | 中等偏高 |

##### 行为模式.json

```json
{
  "已知模式": [
    { "类型": "失败模式", "描述": "约第4周崩溃，原因是目标过大/系统复杂度过高" },

```

**职责**：AI 最重要的智能来源。记录拖延规律、崩溃规律、高效规律、情绪规律。

**现有模式归纳**：

### 4.4 03_执行记录 — 事实层

> 存放每日执行的相关记录，是复盘和动态调整的数据来源。

#### 目录结构

| 目录 | 用途 | 状态 |
|------|------|------|

##### `readJSON(...segments)`

```javascript
function readJSON(...segments) {
  const fp = path.join(ROOT, ...segments);
  if (!fs.existsSync(fp)) return null;
  return JSON.parse(fs.readFileSync(fp, 'utf-8'));
}
```

**职责**：安全的 JSON 文件读取器。
- 拼接路径 → 检查文件存在 → 读取并解析 JSON
- 文件不存在时返回 `null`（不会崩溃）
- 使用 `path.join` 确保跨平台兼容

| 复盘 | 分析趋势、更新观察 | 提供反馈 |

## 9. 关键设计模式与概念

### 9.1 极简优先原则

系统设计中最核心的原则：不要一次性输出巨大系统，逐步构建，每次只推进一个关键点。这直接来源于用户的第 4 周崩溃模式。

### 9.2 动态调整 > 固定计划

没有固定的长期训练计划。AI 根据 5 维状态（精力/动量/纪律/疲劳/压力）和用户反馈实时调整策略。

### 9.3 最低执行模式

当检测到执行断裂信号时，自动切换为只保留最核心 3 项任务，确保系统不崩溃。

### 9.4 持久化记忆

所有关键信息通过文件系统持久化，使 AI 在每次新对话中能完整恢复系统状态，无需依赖历史对话记录。

### 9.5 多层级目标拆解

```
长期目标 (>5年)
  └── 年度目标 (3 Phase)
        └── 月度目标
              └── 周计划
                    └── 每日任务
```

### 9.6 行为模式积累

`行为模式.json` 是 AI 最重要的智能来源。通过持续记录失败模式、风险模式，AI 能够提前预警和制定针对性策略。

## 11. 优化迭代建议

### 11.1 P0 — 数据一致性问题（立即修复）

#### 11.1.1 状态数据不一致

`当前状态.json` 中 `动量: 3, 纪律: 3`，但 `当前人生状态.md` 中显示 `动量: 1, 纪律: 1`。AI系统自身概要.md 中也显示 `动量1 纪律1`。

**建议**：统一数据源，仪表盘 Markdown 文件应从 JSON 自动生成，而非手工维护。

#### 11.1.2 仪表盘文件冗余

`08_仪表盘/` 下同时存在 `本周重点.md` 和 `每周重点.md`，内容不同但命名极易混淆。

**建议**：统一为一个文件 `本周重点.md`，删除 `每周重点.md`。

### 11.2 P1 — 自动化与工程化

#### 11.2.1 仪表盘自动生成

当前 `08_仪表盘/` 下的 Markdown 文件需要手工维护，容易与 JSON 数据源不一致。

**建议**：在 `src/` 下增加一个 `generate_dashboard.js` 脚本，从 JSON 自动生成仪表盘 Markdown 文件。

#### 11.2.2 JSON Schema 校验

6 个核心 JSON 文件没有结构约束，字段漂移风险高。

**建议**：为每个 JSON 定义 schema（可用 JSON Schema 或简单的 TypeScript interface），在 `src/index.js` 中增加校验逻辑。

#### 11.2.3 push.bat 时区问题

`push.bat` 使用 `%date% %time%` 作为 commit message，在不同语言环境下格式不同。

**建议**：改用 `git commit -m "update: $(date '+%Y-%m-%d %H:%M:%S')"`（需 Git Bash），或在 Node.js 中实现推送脚本。

### 11.3 P2 — 功能增强

#### 11.3.1 每日任务自动生成

当前每日任务文件需要手动创建。可以写一个脚本，基于模板 `今日任务.md` 和当前日期自动生成新的每日任务文件。

#### 11.3.2 执行统计面板

在 `src/index.js` 中增加连续执行天数统计、执行率计算等功能，让 `npm run status` 输出更有价值。

#### 11.3.3 补齐预留模块的最小模板

`04_交易系统/`、`06_复盘系统/`、`07_AI观察与动态调整/` 目前全是 `.gitkeep`。为每个模块提供最小可用模板（如复盘模板），降低使用时的心智负担。

#### 11.3.4 复盘模板标准化

当前没有任何复盘模板。建议提供标准化的复盘 Markdown 模板：

```markdown
# 周复盘 — YYYY-MM-DD

## 执行数据
- 执行率: X/7
- 连续执行天数: X
- 状态变化: 精力 X→X, 动量 X→X

## 问题分析
- 主要阻碍:
- 模式匹配:

## 下周优化
- 保留:
- 调整:
- 新增:
```

#### 11.3.5 状态变更历史

当前 `当前状态.json` 只保存最新快照，没有历史记录。

**建议**：在 `07_AI观察与动态调整/长期趋势分析/` 下维护一个状态时间序列，记录每次状态变更，方便追踪长期趋势。

#### 11.3.6 CLI 交互增强

`npm run status` 目前只打印静态信息。可以扩展为支持子命令：

```bash
npm run status           # 当前状态
npm run status -- --log  # 今日执行日志
npm run status -- --week # 本周进度
npm run status -- --help # 帮助
```

### 11.4 P3 — 长期迭代方向

#### 11.4.1 可视化时间线

将状态数据（精力/动量/纪律）绘制为时间线图表，直观展示趋势。

#### 11.4.2 移动端适配

当前系统依赖文件系统和终端，可考虑增加简单的 Web 界面或移动端查看方式。

#### 11.4.3 自动化预警

当连续 N 天未执行或状态指标连续下降时，自动触发预警通知。
