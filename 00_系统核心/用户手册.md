# 零 · 用户手册

> 这份手册只回答一个问题：少爷平时怎么用这个系统最省力。
> 原则：少决策、少翻文件、少解释、少维护。

---

## 一、日常怎么用

### 1. 启动

少爷只需要说：

```text
启动
```

零默认只读取少量必要文件：

```text
00_系统核心/系统总控.md
01_人生核心档案/当前状态.json
05_仪表盘/当前人生状态.md
05_仪表盘/今日任务.md
05_仪表盘/独立执行卡.md
02_目标系统/总体推进计划.md（只引用当前阶段）
```

然后直接输出：

- 当前档位
- 昨夜回顾
- 今日唯一动作
- 晚间底线

### 2. 执行

日常默认只看两个文件：

| 文件 | 用途 |
|---|---|
| `05_仪表盘/今日任务.md` | 看今天完整安排 |
| `05_仪表盘/独立执行卡.md` | 累的时候只看最小动作 |

如果不想看文件，就只听零的简报。

### 3. 汇报

晚上做完后，少爷只需要简短汇报：

```text
训练完成，手机放客厅了
```

或：

```text
今天没做，太累了
```

不用解释太多。零负责记录、判断、调整。

如果完成基线后又多做了，也不用学新口令，直接正常说：

```text
LeetCode 打开了，还做了 3 题
```

或：

```text
训练完成了，今天多做了一组引体
```

零默认把“多做了什么”记为主动推进候选，周复盘再决定是否升级。

如果归档后才想起某项其实完成了，直接说清楚日期和事实即可：

```text
昨天训练其实完成了，也 23:00 前睡了
```

零会把它当成事后补记：先修正当天任务和周计划，再检查是否影响今天的状态、任务和晚间策略。

## 二、维护怎么用

适用场景：改规则、改目标、整理结构、归档历史、检查一致性。

少爷可以直接说：

```text
进入维护
```

或：

```text
维护系统
```

或：

```text
检查一致性
```

维护模式会额外读取：

| 场景 | 读取 |
|---|---|
| 改目标 | `02_目标系统/` |
| 改交易 | `03_交易系统/` |
| 改现金流稳定线 | `04_现金流稳定系统/`（优先看 `现金流稳定线最低启动方案.md` 和 `现金流稳定线文档流计划.md`） |
| 做复盘 | `08_复盘系统/` |
| 改系统规则 | `00_系统核心/系统总控.md` |
| 查历史执行 | `02_目标系统/归档/` |

维护模式默认要做这几类检查：

1. 当前执行区是否混入历史文件。
2. 仪表盘是否只保留当前执行文件。
3. 路径引用是否断链。
4. 数据源和展示层是否重复过多。
5. 事后补记是否已经同步到周计划、当前状态和仪表盘。
6. 结构改动后，总控和手册是否同步。

### 4. 每周最省力复盘法

少爷不用写很长，只要回答 3 个问题：

| 问题 | 作用 |
|---|---|
| 这周最该维持的是哪条线？ | 保住刚稳定下来的东西 |
| 这周最该降级或止损的是哪条线？ | 及时砍掉空转和硬扛 |
| 下周唯一允许的主进阶是什么？ | 防止三条线同时乱加量 |

如果这周做了明显的方向调整，比如：

- 把某条线提成主推
- 把某条线降级
- 暂停某个方向

就在当周周复盘里补一行“决策日志”就够了，不单独再开文件。

## 三、系统现在怎么分层

| 层级 | 目录 | 作用 |
|---|---|---|
| 当前执行层 | `05_仪表盘/` + `02_目标系统/每日任务/` | 放今天/本周真正要用的内容 |
| 规划控制层 | `00_系统核心/` + `01_人生核心档案/` + `02_目标系统/` | 放规则、目标、阶段、数据源 |
| 历史归档层 | `02_目标系统/归档/` + `08_复盘系统/` | 放过往每日任务、复盘、历史执行资料 |

默认理解：

- `05_仪表盘/` 不是仓库，只是当前操作台
- `02_目标系统/每日任务/` 不是历史总库，只放活跃窗口
- 历史仍然保留，但要进归档

## 四、少爷常用指令

| 少爷说 | 零应该做 |
|---|---|
| `启动` | 日常启动，输出简报 |
| `今日任务` | 只输出今天要做什么 |
| `我累` | 自动切保命档或恢复档 |
| `训练完成` | 打勾并更新必要状态 |
| `今天多做了什么` | 记为主动推进候选，留到周复盘决定是否升级 |
| `手机放客厅了` | 记录晚间底线完成 |
| `进入维护` | 进入维护模式 |
| `维护系统` | 从结构/规则/归档角度分析系统 |
| `检查系统` | 检查目录职责与引用 |
| `检查一致性` | 检查路径、镜像、同步链 |
| `复盘` | 进入周/月复盘流程 |
| `结束对话` | 执行结束更新链 |

## 五、使用原则

- 日常不要翻系统，先说 `启动`。
- 累的时候只看 `独立执行卡.md`。
- 状态差时，不补任务，不加任务。
- 状态好时，也只小幅加量。
- 当前执行区不存放历史仓库。
- 中断不是失败，第二天继续最低动作。
- 系统的目标是让少爷越来越强，也越来越便利。

## 六、推送规范

如果零代替少爷执行 git 提交或推送，默认必须遵守：

| 项目 | 规则 |
|---|---|
| 提交格式 | 必须使用 `<type>: <标题>` + 空行 + 详细备注 |
| 详细备注 | 用 3 条左右说明本次改动重点 |
| 发现不规范 | 先修正提交信息，再推送 |
| 依赖来源 | 依赖文件规则，不依赖对话记忆 |

默认示例：

```text
chore: 重构零系统并校准目标流转

- 重构系统分层与归档结构
- 校准核心档案与目标系统
- 补齐推演、6月目标与统一推进规则
```
